# Skill: content-review

## 用途

对生成的小红书笔记进行强制质量检查，确保符合人设要求、结构规范和平台合规性。

## 调用方式

```
skill: content-review
内容：[需要检查的笔记全文]
批次状态：[本次批次已使用的元素JSON，可选]
```

## 检查流程

### 第一步：人设一致性检查

**检查清单**：
- [ ] 年限是否为"8年"？（出现"10年""多年"等模糊表述即失败）
- [ ] 背景是否为"ALMA厨艺学院"？（出现"蓝带""法国"等其他背景即失败）
- [ ] 人称是否全程使用"我"？（出现"我们""咱们"即失败）
- [ ] 是否出现禁用品牌词？（"一碟盐"等即失败）

**处理方式**：
- 任何一项失败 → 返回失败，要求返工
- 提供具体位置（行号）和错误原文

### 第二步：4步结构检查

**检查清单**：
- [ ] 是否完整包含4个步骤？
- [ ] 第1步（痛点）是否3-5句话？（充分展开场景）
- [ ] 第2步（观点）是否3-5句话？（充分展开）
- [ ] 第2步（观点）是否回应了前文痛点？
- [ ] 第2步（观点）后是否有具体例子支撑？
- [ ] 是否跳过任何一步？
- [ ] 顺序是否为1→2→3→4？

**结构识别方法**：
1. **痛点识别**：查找关键词"见过太多""反馈不好干""发现""问题在于"等
2. **观点识别**：查找"说实话""我发现""记住""其实""说到底""回过头看"等观点引导词
3. **案例识别**：查找"给一家""上周""之前"等服务场景描述
4. **结尾识别**：查找"评论区""聊聊""记住"等互动或总结词

**处理方式**：
- 步骤缺失或顺序错误 → 返回失败
- 痛点少于3句或超过5句 → 返回失败，要求调整
- 观点少于3句或超过5句 → 返回失败，要求展开
- 观点未回应痛点 → 返回失败，要求修正
- 观点后无具体例子 → 返回失败，要求补充

### 第三步：菜品命名检查

**检查清单**：
- [ ] 是否照搬素材库原菜名？（对比素材库文件）
- [ ] 菜名是否有高级感和创新性？
- [ ] 菜名是否过长（>10字）？
- [ ] 是否使用了简单堆砌（多个食材名直接拼接）？

**参考素材库**：
- 素材库/笔记素材/笔记1-提升40%研发融合料理底层逻辑.md
- 素材库/笔记素材/笔记2-融合料理培训16道年夜漂亮饭客流拉满.md
- 素材库/笔记素材/笔记3-bistro破局融合料理才是出路.md
- 素材库/笔记素材/笔记4-Bistro春季菜单换新.md
- 素材库/笔记素材/笔记5-融合料理培训复购好不好80%看酱汁.md
- 素材库/笔记素材/笔记6-为上海bistro做了一道原创融合料理.md

**处理方式**：
- 发现原样照搬 → 返回失败，要求重新命名
- 菜名缺乏创意 → 返回警告，建议优化
- 菜名过长 → 返回失败，要求精简

### 第四步：违禁词检查

**检查清单**：

**高危词（必须修改）**：
- 直白价格："成本X块卖X""8块卖48"
- 绝对化："绝对""最好""第一""顶级"
- 夸张宣传："稳赚""包赚""必火"
- 医疗功效："养生""保健""疗效"

**中危词（建议修改）**：
- 营销话术："包教包会""加持""一站式解决"
- 导流诱导："加微信""私信我""进群"
- 虚假稀缺："限量""仅剩""手慢无"

**新增敏感词**：
- 业绩夸张："翻倍""暴涨""暴增""暴涨""流量暴涨"
- 引流相关："引流""导流""拉新"
- 过度承诺："轻松月入""零基础也能""小白也能"

**处理方式**：
- 发现高危词 → 返回失败，提供替代表述
- 发现中危词 → 返回警告，建议修改

### 第五步：观点衔接检查

**检查清单**：
- [ ] 观点是否回应了前文的痛点？
- [ ] 观点与前文是否有割裂感？
- [ ] 观点后是否跟了具体例子支撑？
- [ ] 观点是否使用了6种方式之一（说实话/我发现/记住/其实/说到底/回过头看）？
- [ ] 观点方式是否与上1篇重复？

**观点回应检查**：
- 如果痛点是"菜品研发难"，观点应给出研发方向
- 如果痛点是"招不到好主厨"，观点应讲标准化
- 如果痛点是"同质化严重"，观点应讲差异化

**例子支撑检查**：
- 观点后必须有1个具体例子（菜品/数据/场景）
- 例子要具体，不能泛泛而谈

**处理方式**：
- 观点未回应痛点 → 返回失败，要求重写观点
- 观点与前文割裂 → 返回失败，要求调整过渡
- 观点后无例子 → 返回失败，要求补充
- 观点方式重复 → 返回警告，建议更换

### 第六步：跨篇去重检查（如有批次状态）

**检查清单**：
- [ ] 开场白变体是否与已使用的重复？
- [ ] 观点方式是否与已使用的重复？（至少隔2篇）
- [ ] 创新角度是否与已使用的重复？
- [ ] 素材组合是否与已使用的重复？
- [ ] 菜品名称是否与已使用的重复？

**处理方式**：
- 发现重复 → 返回失败，要求更换

### 第七步：语气风格检查

**检查清单**：
- [ ] 是否过于营销化？（满篇"✅""🔧""加持"等）
- [ ] 是否符合"掏心窝子聊天"风格？
- [ ] 是否出现说教感？

**处理方式**：
- 过于营销 → 返回失败，要求改写
- 轻微偏差 → 返回警告

## 使用示例

**输入**：
```
skill: content-review
内容：
做餐饮10年，蓝带主厨，见过太多餐厅在菜单上栽跟头。

说实话，春季菜单的核心不是堆砌时令食材，而是找到平衡。

我们给一家餐厅做了菜单升级，效果很好。

👉扣【菜单】获取方案！

批次状态：
{
  "used_opening_variants": ["变体2"],
  "used_dish_names": ["春日绯红蟹肉塔"]
}
```

**输出**：
```markdown
## 内容审查报告

### 检查结果：❌失败

**总体评分**：45/100

### 详细检查项

#### 1. 人设一致性 ❌失败
- 年限检查：❌失败（出现"10年"，应为"8年"）
- 背景检查：❌失败（出现"蓝带"，应为"ALMA"）
- 人称检查：❌失败（出现"我们"，应为"我"）

#### 2. 4步结构 ✅通过
...

### 问题详情

**严重问题（必须修改）**：
1. 人设年限错误 → 第1行"做餐饮10年" → 改为"餐饮8年"
2. 人设背景错误 → 第1行"蓝带主厨" → 改为"ALMA厨艺学院毕业"
3. 人称错误 → 第5行"我们给一家餐厅" → 改为"我给一家餐厅"

### 最终判定

- **是否通过**：否
- **是否可发布**：否
- **需要返工**：是
- **返工重点**：修正人设一致性（年限、背景、人称）
```

## 注意事项

1. **强制拦截**：任何一项严重问题都导致整体失败
2. **明确位置**：必须指出具体行号和原文
3. **提供建议**：不仅指出问题，还要提供修改方案
4. **批次联动**：如有批次状态，必须进行跨篇去重检查
5. **记录问题**：将常见问题汇总，用于后续优化提示词
